# 管理端功能接口文档

本文档基于当前前端项目已封装的 API 与已落地页面整理，描述管理端现有接口、页面能力和实现状态。

## 1. 通用约定

### 1.1 鉴权

除登录、注册、刷新 Token 等认证接口外，业务接口均通过 JWT Bearer Token 鉴权。

```http
Authorization: Bearer <accessToken>
```

前端通过 `src/utils/request.ts` 统一注入 Token，并在 401 且存在 refreshToken 时自动调用刷新接口。

### 1.2 统一响应结构

```ts
interface ApiResponse<T> {
  code: number
  message: string
  data: T
}
```

前端通过 `unwrapApiResponse` 统一处理响应，`code !== 200` 时抛出错误。

### 1.3 分页响应结构

```ts
interface PageData<T> {
  items: T[]
  page: number
  pageSize: number
  total: number
  hasMore: boolean
}
```

## 2. 认证模块

实现页面：

- 登录页
- 注册页
- 退出登录
- Token 自动刷新

接口清单：

| 功能 | 方法 | 接口 | 说明 |
| --- | --- | --- | --- |
| 登录 | POST | `/admin/auth/login` | 使用手机号和密码登录，返回 accessToken、refreshToken 和用户信息 |
| 注册 | POST | `/admin/auth/register` | 注册管理端账号，支持 `ADMIN`、`HOUSEKEEPER`、`LANDLORD` 角色 |
| 刷新 Token | POST | `/admin/auth/refresh` | accessToken 失效后使用 refreshToken 换取新 Token |
| 退出登录 | POST | `/admin/auth/logout` | 提交 refreshToken，服务端注销登录态 |

已实现功能：

- 登录成功后保存 accessToken、refreshToken 和当前用户信息。
- 页面刷新后恢复本地登录态。
- 请求遇到 401 时自动刷新 Token 并重试原请求。
- 退出登录时调用服务端接口，同时清理本地登录态。

### 3.1 房源列表

| 功能 | 方法 | 接口 | 说明 |
| --- | --- | --- | --- |
| 获取房源列表 | GET | `/admin/houses` | 获取房源基础信息、运营状态、门锁绑定摘要 |

已实现功能：

状态枚举：

| 状态值 | 展示文案 | 说明 |
| --- | --- | --- |
| `draft` | 草稿 | 新增后未发布的房源 |
| `available` | 可租 | 已发布且可对外展示 |
| `reserved` | 已被预定 | 已被租客预定 |
| `rented` | 已租 | 已出租 |
| `offline` | 下架 | 不再对外展示 |

## 10. 当前已实现的关键业务闭环

### 10.1 房源运营闭环

1. 新增房源并上传图片。
2. 房源以 `draft` 草稿状态进入列表。
3. 管理员可编辑房源信息。
4. 草稿房源可发布。
5. 草稿或可租房源可下架。
6. 房源列表展示当前状态和门锁绑定摘要。

### 10.2 门锁绑定闭环

1. 通过 MAC 查询或录入本地初始化门锁。
2. 将门锁绑定到房源或房间。
3. 查看门锁详情、平台同步状态和蓝牙状态。
4. 更新蓝牙状态。
5. 获取开锁数据。
6. 必要时解除绑定。

### 10.3 退租处理闭环

1. 查询退租申请列表。
2. 查看退租详情。
3. 审核通过、驳回或要求补充材料。
4. 结算阶段确认扣款和退款金额。
5. 完成退租流程。

## 11. 后续建议

- 为房源详情补充独立详情接口和完整编辑页，减少列表行数据不完整时的编辑风险。
- 房源图片编辑建议复用新增页的上传组件，避免手动维护 URL。
- 将房源状态、租赁类型、退租状态抽成统一常量，避免页面内重复维护。
- 为用户、订单、账单、报修、系统设置接入真实后端接口。
- 为房源列表增加服务端分页和筛选，避免房源数量增大后前端分页性能下降。
